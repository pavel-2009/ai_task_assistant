# Portfolio Test Plan

Документ фиксирует полный набор тестов, который нужно иметь в проекте, чтобы его можно было уверенно показывать в портфолио.

---

## 1) Базовый минимальный уровень (обязательный)

### 1.1 Unit-тесты (быстрые, изолированные)

- `auth`:
  - хеширование/проверка пароля;
  - создание/валидация JWT;
  - обработка истёкшего токена.
- `core/security`:
  - отказ при невалидных credentials;
  - корректная работа helper-функций безопасности.
- `utils/math_ops`:
  - happy-path и edge-cases (0, отрицательные, большие значения).
- `ml/nlp/vector_db`:
  - upsert/search/delete для документа;
  - пустой индекс, дубликаты id.
- `ml/recsys/content_based`:
  - сортировка рекомендаций по score;
  - отсутствие self-recommendation, если требуется контрактом.
- `ml/recsys/collaborative_filtering`:
  - строгий контракт сериализации (save/load);
  - корректная форма матриц/векторов и размерности на инференсе;
  - fallback на popular items для cold-start.

### 1.2 API-тесты (роуты FastAPI)

- `POST /auth/register`, `POST /auth/login`:
  - успешный сценарий;
  - дубликат пользователя;
  - неверный пароль.
- `GET/POST/PUT/DELETE /tasks`:
  - полный CRUD;
  - доступ только для владельца;
  - валидация входных данных.
- `GET /ping`:
  - возвращает `status=ok` и валидную структуру ответа.

### 1.3 Интеграционные тесты (обязательные для портфолио)

- Auth + Tasks:
  - регистрация -> логин -> создание задач -> чтение списка от имени того же пользователя.
- DB + API:
  - миграции применяются, после чего API работает на чистой БД.
- RecSys pipeline:
  - подготовка interactions -> обучение -> сохранение в Redis/хранилище -> загрузка -> `recommend()`.
- NLP RAG pipeline:
  - индексация документов -> retrieval -> генерация ответа (LLM мокается, retrieval проверяется реально).
- CV inference pipeline:
  - загрузка изображения -> классификация/детекция endpoint -> валидный JSON-ответ с полями предсказания.

---

## 2) Расширенный уровень (сильно повышает качество портфолио)

### 2.1 Контрактные тесты (schema/serialization)

- Стабильность JSON-схем API (snapshot или pydantic schema assert).
- Контракт Celery task payload/result.
- Контракт сериализации моделей RecSys/NLP (совместимость между версиями).

### 2.2 E2E smoke (docker-compose)

Запуск в CI или локально в контейнерах:

1. `docker-compose up -d`.
2. Ожидание readiness (`/ping`).
3. Smoke-сценарий:
   - register/login;
   - create task;
   - вызвать один NLP endpoint;
   - вызвать один CV endpoint;
   - получить рекомендации.
4. `docker-compose down -v`.

### 2.3 Нагрузочные/устойчивость

- Smoke-load на ключевые endpoint (например, `tasks`, `ping`, `recsys`).
- Проверка rate limit под burst-нагрузкой.
- Проверка graceful shutdown (в процессе обработки запросов).

---

## 3) Что должно быть в CI (минимум)

- `ruff` / `flake8` (стиль).
- `black --check` (форматирование).
- `pytest -q` (unit + integration).
- отдельная job для docker smoke.
- публикация покрытия (`pytest --cov=app`) и порог не ниже 70% (целевой 80%+).

---

## 4) Тесты, которые стоит написать в первую очередь (приоритет)

1. **RecSys serialization + recommend roundtrip** (критично для корректности рекомендаций).
2. **Интеграционный Auth->Tasks flow** (базовый пользовательский сценарий).
3. **RAG integration с мокнутым LLM** (дешево и стабильно в CI).
4. **CV endpoint smoke на одном тестовом изображении**.
5. **Docker E2E smoke**.

---

## 5) Definition of Done для портфолио

Проект можно считать готовым к показу в портфолио, если одновременно выполнены условия:

- unit-тесты покрывают ключевую бизнес-логику auth/tasks/recsys/nlp/cv;
- есть интеграционные тесты для сквозных пользовательских сценариев;
- есть docker-smoke e2e;
- тесты стабильно проходят в CI на чистом окружении;
- документированные команды запуска тестов работают без ручных «доправок» окружения.
