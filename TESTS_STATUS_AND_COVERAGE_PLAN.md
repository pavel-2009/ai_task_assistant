# Оценка текущего состояния тестов и план покрытия

## 1) Что уже покрыто

По структуре репозитория видно, что есть как минимум два слоя тестов:

- API-тесты (`tests/api/*`) для:
  - аутентификации,
  - задач,
  - NLP,
  - загрузки аватаров.
- Unit-тесты (`tests/unit/*`) для:
  - auth/security,
  - NLP,
  - Celery-задач,
  - создания задач,
  - upload-логики.

Также есть общий `tests/conftest.py` с тестовой SQLite-конфигурацией и базовыми фикстурами пользователей/токенов.

## 2) Текущие риски по качеству тестов

1. **Нестабильность кодов ответа (422 vs 400)**
   - Для части auth-кейсов поведение может отличаться между окружениями (локально/CI), особенно для валидации form/json payload.
   - Это типичная проблема различий версий FastAPI/Starlette/Pydantic и порядка обработки исключений.

2. **Сильная связность с интеграционными деталями**
   - Часть тестов зависит от состояния rate limiter / времени жизни токена.
   - Такие тесты чувствительны к скорости окружения и могут флапать.

3. **Ограниченное покрытие “сквозных” сценариев**
   - Сейчас видно хорошее покрытие отдельных эндпоинтов, но меньше end-to-end флоу (регистрация → логин → создание задач → NLP/RAG → фоновые задачи).

## 3) Что критично дополнительно протестировать для должного покрытия

### 3.1 Ядро API и безопасность

- Полный lifecycle JWT:
  - выпуск токена,
  - проверка срока,
  - обработка невалидного/подменённого токена,
  - поведение protected endpoints при отсутствии токена.
- Граничные кейсы rate-limit:
  - reset окна,
  - конкурентные запросы,
  - независимость лимитов разных маршрутов.

### 3.2 Модуль задач (`routers/tasks`, `services`, `schemas/task`)

- Авторизационная изоляция данных между пользователями во всех CRUD-операциях.
- Валидация полей `title/description` на границах (min/max длины, пустые значения).
- Корректность пагинации/фильтрации (если применимо).

### 3.3 NLP/RAG (`routers/nlp`, `routers/rag`, `ml/nlp/*`)

- Контрактные тесты входов/выходов API (включая пустые и невалидные payload).
- Тесты отказоустойчивости внешних зависимостей (timeout/ошибки LLM API) с моками.
- Проверка консистентности поиска/ранжирования (на фиксированных тестовых данных).

### 3.4 CV и загрузка изображений (`routers/avatars`, `ml/cv/*`, `utils/image_ops`)

- Набор негативных тестов:
  - не-изображение,
  - слишком большой файл,
  - повреждённый файл,
  - неподдерживаемый mime.
- Unit-тесты preprocessing/postprocessing для CV-инференса на маленьких фикстурах.

### 3.5 Celery/фоновые задачи (`celery_app`, `ml/*/tasks.py`)

- Smoke-тесты регистрации Celery-task names.
- Интеграционные тесты постановки задач в очередь (с тестовым Redis/моком broker).
- Проверка beat-расписания: наличие и корректность ключевых periodic задач.

### 3.6 Мониторинг и ошибки (`error_handlers`, `core/metrics`, `routers/monitoring`)

- Проверка единого формата ошибок (`code`, `detail`) на основных типах исключений.
- Тесты `/metrics` и health/ping endpoint.
- Проверка, что 5xx логируются и корректно маскируют внутренние детали.

## 4) Практический план

1. Зафиксировать версии FastAPI/Starlette/Pydantic и стабилизировать спорные тесты на 422/400.
2. Ввести маркеры для быстрых/интеграционных/медленных тестов и выполнять их раздельно в CI.
3. Добавить отдельный nightly job для тяжёлых ML/CV/NLP сценариев.
4. Внедрить отчёт покрытия (`pytest-cov`) и целевой порог (например, 75%+, затем повышать).

## 5) Минимальный критерий готовности

- Стабильный зелёный CI на PR.
- Покрытие критических потоков auth + tasks + базовых ошибок API.
- Отдельные smoke-тесты Celery и проверка конфигурации beat.
- Чёткий список исключений/xfail с объяснением причин и планом устранения.
